# UI/UX 예외 처리 및 재발방지 가이드 (Post-Mortem & Guidelines)

## 1. 개요 (Incident Summary)
- **발생 이슈**: 대시보드의 '주의 알림 (CHECK)' 섹션 내 필터에서 특정 항목('결제 확인') 선택 시, 필터 UI를 포함한 알림 박스 전체가 예고 없이 화면에서 사라지는 현상.
- **결과 및 불편사항**: 조작 패널(필터 셀렉트박스)이 통째로 증발하여, 이용자가 다른 조건으로 다시 검색하거나 화면을 원래대로 복구할 수 없는 'Dead End(막다른 길)' 상태에 빠짐.

## 2. 근본 원인 (Root Cause)
1. **Empty State(빈 화면)에 대한 잘못된 렌더링 방식**: 
   해당 조건에 부합하는 대상자 데이터가 0명일 때, 사용자에게 "데이터가 없음"을 시각적으로 안내하는 대신, 해당 컴포넌트의 영역 자체를 숨김 처리(`display: none`)하는 원시적인 코드가 적용되어 있었습니다.
2. **UI의 논리적 종속 관계 오류**: 
   사용자의 '조작(Control)'을 담당하는 필터가 '결과(Result)'를 보여주는 박스 내부에 종속되어 있었습니다. 결과값이 빈칸이라는 이유로 상위 박스를 지워버리니, 조작 도구마저 파괴되었습니다.

## 3. 재발 방지 대책 및 개발 원칙 (Preventive Actions)

### A. 모든 영역에 'Empty State' 표준 규격화
모든 테이블, 리스트, 알림 컴포넌트는 **데이터가 없을 때(0건) 어떻게 보일지(Empty State)를 사전에 정의하고 렌더링**해야 합니다.
- ❌ **금지**: 화면/영역을 통째로 숨겨버려 화면 구조를 붕괴시키는 행위.
- ✅ **권장**: 레이아웃 골격은 그대로 둔 채, 내부 컨텐츠 자리에만 "조회된 내역이 없습니다", "조건에 맞는 멤버가 없습니다"와 같은 친절한 안내 텍스트 노출.

### B. 조작부(Control)와 결과부(View)의 완전한 분리 보장
**사용자가 언제든지 다른 조건을 재탐색할 수 있도록, 제어 도구(Filter, Search Bar, Tab)는 결과 화면의 유무에 관계없이 독립적으로 항상 살아있어야 합니다.**
- 컴포넌트 설계 시 조작부가 들어간 Header와 데이터가 나오는 Body 영역을 구조적으로 명확히 분리하여 렌더링 간섭을 차단하겠습니다.

### C. 코너 케이스(Corner Case) 검증 프로세스 도입
이번 사태처럼 "데이터가 아예 존재하지 않는" 극단적 상황을 대비하기 위해 프론트엔드 작업 시 다음 사항을 필수로 테스트하겠습니다.
- 아무런 데이터가 없을 때 UI가 어떻게 보이는가? (Zero-data edge case)
- 데이터나 이름이 비정상적으로 길 때 박스가 깨지지 않는가? (Overflow test)
- 모든 탭과 필터가 상호작용하더라도 화면 레이아웃(크기, 스크롤)이 안정적인가?
